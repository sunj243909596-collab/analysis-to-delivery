# V1/V2 双版本工作流

> 适用于阶段 8（开发设计）
> V1 = 存储过程 / 详细设计版，V2 = 代码实现版

## 为什么需要双版本

- **V1**：业务逻辑精准表达，便于评审、SQL 调优、问题定位
- **V2**：完整代码实现，含前后端、联调、Mock 数据

两者**不重复**：V1 关注"做什么"，V2 关注"怎么实现"。

## V1 内容要求

- PL/SQL 包清单
- 包规范 + 包体实现
- 核心 SQL
- 错误码定义
- tXML 接口
- 部署清单

输出物：`02-功能规格说明书 FSD.md` + `03-数据模型设计.md`

## V2 内容要求

- 后端分层（Controller / Service / Repository / Entity）
- Service 实现
- Mapper XML
- DTO / VO
- REST API
- 事务异常处理
- **前端组件**（PC + APP，按需）
- 联调说明
- Mock 数据
- Checklist

输出物：`06-开发设计说明书.md`

## 关键纪律

### 严禁一次生成两个版本

```
V1 → 用户确认 → V2
      ↑
   必须等用户审阅
```

### V2 严禁降级

- 遇错重试
- 不允许跳过
- 不允许简化为"伪代码"

### V2 前后端完整性

仅含后端 Service/Mapper 视为**不完整**。必须含：
- Controller 层
- PC 端 Vue 组件
- APP 端 uni-app 组件（如适用）
- 联调步骤
- Mock 数据
- 错误码映射

## 双版本对照验证

V1 → V2 后，对照检查：
- 业务逻辑一致
- 错误码一致
- 数据写入一致
- PIX 记账一致
- 事务原子性一致
- SQL 查询一致

不一致 → 修正 → 重跑对照
