# 开发设计规范（v1.1）

> 适用于阶段 8（开发设计）。
> v1.1 起去掉 V1（存储过程版）/ V2（代码实现版）双版本概念，统一为单一代码版工作流。

## 阶段 8 三件套

阶段 8 产出**三个文档**，按以下顺序生成：

```
FSD（功能规格）→ 数据模型设计 → 开发设计说明书
   ↑               ↑                  ↑
  8.1             8.2                8.3
```

| 文档 | 作用 | 输入 | 输出 |
|------|------|------|------|
| `02-功能规格说明书 FSD.md` | 功能视角：模块/接口/状态/错误 | PRD + TASK_CONFIRM | "功能怎么用" |
| `03-数据模型设计.md` | 数据视角：表/字段/索引/序列 | FSD | "数据怎么存" |
| `06-开发设计说明书.md` | 技术视角：分层/接口/前端/联调 | FSD + 数据模型 | "代码怎么写" |

## 8.1 FSD 内容要求

（参考 `templates/FSD.md`）

- 模块清单
- 功能详述（输入/输出/业务规则/边界/异常）
- 接口契约（REST / RPC / 消息，含入参出参错误码）
- 状态机（关键业务对象的状态流转）
- 数据模型概览（哪些表、哪些字段、哪些关联）
- 业务错误码字典

**FSD 与 BRD 的区别**：
- BRD：业务视角，回答"为什么要做"
- FSD：功能视角，回答"功能怎么用"
- 同样一份需求，BRD 给业务方看，FSD 给开发方看

## 8.2 数据模型设计内容要求

（参考 `templates/数据模型设计.md`）

- 表清单（新增 / 扩展）
- 表结构 + 索引 + 序列
- 字段对齐验证（与 `knowledge-path.md` 引用的知识库核对）
- 初始化数据
- 迁移脚本（如适用）

## 8.3 开发设计说明书（代码版）内容要求

（参考 `templates/开发设计说明书.md`）

- 架构设计（分层、关键依赖）
- 后端分层（Controller / Service / Repository / Entity / Mapper XML / DTO/VO）
- 事务与异常处理
- **前端组件**（PC + APP，按需）
- 联调说明（接口清单、步骤、Mock 数据）
- 错误码映射
- Checklist

## 关键纪律

### 严禁只写后端

仅含后端 Service/Mapper 视为**不完整**。必须含：
- Controller 层
- PC 端 Vue 组件
- APP 端 uni-app 组件（如适用）
- 联调步骤
- Mock 数据
- 错误码映射

### 严禁降级

- 遇错重试
- 不允许跳过
- 不允许简化为"伪代码" / "TODO 列表"
- 不允许"留待开发时定"

### 字段对齐

- 字段名 / 表名 / 状态码必须与 `knowledge-path.md` 引用的知识库一致
- 业务同义不同名需在注释中显式标注
- 严禁自行发明字段 / 表名

### PL/SQL 包 / 专有消息协议接口（如确需）

不再单独成文档，统一**内嵌**到：
- FSD 的"接口契约 → RPC / 消息"小节（适用于专有消息协议）
- 开发设计说明书的"事务与异常处理"小节（适用于 PL/SQL 包）

## 实战教训

- 2026-04-27 收货管理 V2：审查发现 SKILL 5.5.4 过度聚焦后端，缺失 Controller/前端/联调。V2 文档从 2375 行补充至 3000+ 行
- v1.1 简化：用户反馈"开发设计直接代码版就行，存储过程不需要" → 去掉双版本，工作流更轻

## 与旧版 V1/V2 流程的差异

| 维度 | v1.0 V1/V2 双版本 | v1.1 单一代码版 |
|------|------------------|-----------------|
| 阶段数 | 2 个子阶段（V1 评审 → V2 出文） | 3 个子阶段（FSD → 数据模型 → 开发设计） |
| 中间产物 | 单独成文档的 PL/SQL 包 | 内嵌到 FSD / 开发设计 |
| 评审节奏 | 必须等 V1 确认才能出 V2 | 顺序出 3 个文档，无需中间停顿 |
| 适用人群 | 复杂存储过程 / SQL 调优场景 | 大多数应用层项目 |
| 出口 | `02-FSD.md` + `03-数据模型.md` + `06-开发设计.md` | 同上 |
