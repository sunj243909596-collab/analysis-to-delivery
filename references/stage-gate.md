# 阶段门控（Stage Gate）

> 适用于本 skill 的所有 10 阶段及阶段 8 内的开发实施子流程。
> 目标：**任何一步没签字，就不能进下一步**。

## 三层门控

按粒度从粗到细分三层。每一层都有自己的"通过标准"和"签字方"。

```
┌─────────────────────────────────────────────────┐
│ 第 1 层：工作流门控（10 阶段之间）              │  ← 用户签字
├─────────────────────────────────────────────────┤
│ 第 2 层：开发子流程门控（阶段 8 内）            │  ← 用户 + Claude 双向确认
├─────────────────────────────────────────────────┤
│ 第 3 层：子任务门控（writing-plans 的每个任务）│  ← 测试 + 用户抽查
└─────────────────────────────────────────────────┘
```

---

## 第 1 层：工作流门控（粗粒度）

适用：10 阶段之间的 9 个过渡点。

| 过渡点 | 通过标准 | 签字方 |
|---|---|---|
| 1 → 2（需求澄清 → BRD） | TASK_CONFIRM + REVIEW_需求确认书 已签字 | 用户 |
| 2 → 3（BRD → 合规） | BRD §1-9 必备章节齐 | 用户 |
| 3 → 4（合规 → 系统方案） | 合规评审结论为 ✅ 通过 / ⚠️ 带条件通过 | 用户 + 合规方 |
| 4 → 5（系统方案 → 测试用例） | 系统方案已确定数据走向 + 状态机 | 用户 |
| 5 → 6（测试用例 → PRD） | 测试用例覆盖主流程 + 异常 | 用户 |
| 6 → 7（PRD → 委派） | PRD §七验收标准签字 | 用户 |
| 7 → 8（委派 → 开发设计） | 委派的 Claude 子代理已交付 | 用户 |
| 8 → 9（开发设计 → QA 审计） | 设计回测已通过（见 design-backtest.md）| 用户 |
| 9 → 10（QA 审计 → 交接） | `full-qa-audit.py` P0=0 | 用户 + 接收方 |

### 实现方式

每个文档模板顶部加一个"门控签字区"：

```markdown
## 阶段门控

| 项目 | 状态 | 签字 / 时间 |
|---|---|---|
| 上一阶段签字 | ⬜ 未签字 / ✅ 已签字 | （人名 + 日期） |
| 本阶段交付物完整性 | ⬜ 缺 / ✅ 齐 | 自检 |
| 下一阶段就绪 | ⬜ 否 / ✅ 是 | （人名 + 日期） |
```

> **HARD GATE**：上一阶段未签字，禁止进入下一阶段。Claude 不得自动跨门。

## 第 2 层：开发子流程门控（细粒度）

适用：阶段 8 内的 5 个子流程（brainstorming / spec / plan / TDD / execute）。

| 子流程门 | 通过标准 | 签字方 |
|---|---|---|
| brainstorming → spec | 设计稿写到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`，**用户明确批准** | 用户 |
| spec → writing-plans | spec 内联自检通过（无占位符 / 无矛盾 / 范围明确）| 用户 |
| writing-plans → TDD | 计划任务粒度 ≤ 2 小时 / 任务，文件路径 + 验证命令齐全 | 用户 |
| TDD → execute | RED 测试已写完、已运行确认失败 | 用户抽查 |
| execute → 下一任务 | 当前任务 GREEN + REFACTOR + commit | Claude 自检 + 用户抽查 |

### 实现方式

阶段 8 文档模板（`06-开发设计说明书.md`）"八、Checklist" 之后加一节"九、子流程门控清单"：

```markdown
## 九、子流程门控清单

| 子流程 | 状态 | 签字 / 时间 |
|---|---|---|
| brainstorming 设计稿 | ⬜ | |
| spec 自检 | ⬜ | |
| writing-plans 完成 | ⬜ | |
| TDD RED 测试 | ⬜ | |
| 全部子任务 GREEN | ⬜ | |
```

## 第 3 层：子任务门控（最细粒度）

适用：writing-plans 拆出的每个 ≤ 2 小时任务。

每个子任务的"完成"必须满足三件事：

1. **代码层**：
   - 测试已写 + 已 RED + 已 GREEN
   - 验证命令已跑（按 plan 中指定的命令）
   - 改动已 commit（commit message 含任务编号）
2. **文档层**：
   - 任务复盘已写（见 task-retrospective.md）
3. **沉淀层**：
   - 发现的"通用知识"已追加到 `knowledge-path.md` 指向的知识库

### 实现方式

子任务用 TodoWrite / TaskCreate 跟踪。每个任务状态：

- ⬜ pending（未开始）
- 🟡 in_progress（进行中）
- 🔴 RED（测试写完，已确认失败）
- 🟢 GREEN（代码完成，测试通过）
- ✅ completed（已 commit + 已写复盘 + 已沉淀）

> **不要跳状态**。RED 没确认就写实现 = 假 TDD。

---

## 跳过门控的合法理由

以下情况允许**有条件跳过**（但必须记录在阶段 8 文档"九、子流程门控清单"中）：

| 理由 | 适用 | 怎么做 |
|---|---|---|
| 紧急修复（hotfix） | 阶段 8-10 | 事后补签字；hotfix PR 描述里写明"门控后补" |
| 纯配置变更（不改代码） | 阶段 8 | 第 3 层子任务门控可省；前两层仍要 |
| 极小项目（单函数 / 1-2 文件） | 阶段 8 全流程 | 第 2 层可压缩为 1 步（brainstorming → execute 直跳），但用户必须明确批准 |
| 探索性 spike | 阶段 8 | 第 2 层可省；事后决定是否进入正式开发 |

## 与 superpowers 的对应

本门控机制是 superpowers `<HARD-GATE>` 的项目级延伸：

| superpowers 概念 | 本 skill 的落地 |
|---|---|
| `<HARD-GATE>` 设计稿未批准不许实现 | 第 2 层 brainstorming → spec 门 |
| TDD "watch it fail" 原则 | 第 3 层 RED 必须显式确认 |
| Subagent 两阶段 review | 第 3 层任务完成的"代码 / 文档 / 沉淀"三段式 |

> 本 skill 不重复 superpowers 的内容，只把它的纪律**嵌入到项目级交付物**里。
