# 字段对齐方法论

> 适用于：任何涉及现有表查询/维护类功能的设计文档

## 核心理念

**任何涉及现有表的设计文档（PRD/FSD/数据模型）前，必须先建立"需求字段 ↔ 知识库实际表名+字段"的映射关系。**

## 流程

### Step 1: 提取需求中的表名和字段
- 从 `TASK_CONFIRM_*.md` 提取
- 从 `REVIEW_需求确认书.md` 提取
- 从用户的对话中提取

### Step 2: 核对知识库

对每个表和字段：
1. 搜索 `config/domain-knowledge/{领域}.md` 引用的知识库文件
2. 找到真实表名（不是猜测的）
3. 逐字段核对：字段名、类型、长度、可空性

### Step 3: 输出对齐报告

```
| 知识库定义字段 | 类型 | 可空 | PRD 中的引用 | 对齐状态 |
```

4 种状态：
- ✅ 已对齐
- ⚠️ 需 JOIN（标注关联表）
- ❓ 待确认（业务同义不同名）
- 🔴 缺失

### Step 4: 用户确认
- 输出 `REVIEW_字段对齐分析.md`
- 用户审阅 → 确认 → 才能进入阶段 2

## 业务同义不同名

**问题**：PRD 写"订单组号=ASN.REF_FIELD_4"，但知识库中 REF_FIELD_4 实际定义是"货主订单号"。

**处理**：
1. 与用户确认是否业务同义
2. 同义 → PRD 注释里显式标注：
   ```
   ASN.FTSR_NBR (货主订单号 = 业务上的订单组号)
   ```
3. 不同义 → 找真正的字段，或询问用户是否需要新增字段

## 严禁自创字段

- ❌ 不要写 `BIZ_FLAG`、`EXT_FIELD_X`（除非知识库允许）
- ✅ 用 `REF_FIELD_1` ~ `REF_FIELD_5`（多数系统预留的扩展位）
- ❌ 严禁修改知识库已定义的字段含义或类型

## 委派 Claude 时的硬性约束

委派生成文档时，必须显式列出每个表的真实表名和真实字段清单：

```
表名：ASN（不是 TC_ASN_HDR）
真实字段：ASN_ID(NUMBER), TC_ASN_ID(VARCHAR2(50)), ASN_STATUS(NUMBER(3)), ...

表名：ASN_DETAIL（不是 TC_ASN_DTL）
真实字段：ASN_DETAIL_ID(NUMBER), SKU_ID(NUMBER), BATCH_NBR(VARCHAR2(200)), ...
```

Claude 只能在上述清单内选择，不得自行发明。

## 计算字段标注

如"剩余可收数量"等计算字段，必须标注为：
```
QTY_REMAINING = QTY_EXPECTED - QTY_RECEIVED  -- 计算字段，非数据库列
```

不能写成数据库列。

## 实战教训

- 2026-05-19 收货简化版：跳过此步 → 16 个字段错误
- 2026-06-09 随货同行单：业务同义未标注 → 用户审阅发现
- 2026-04-30 序列号追溯：PIX_TYPE 字符串误用 → 改成 VARCHAR2(3) 数字码
