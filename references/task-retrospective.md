# 任务复盘（Task Retrospective）

> 适用于 writing-plans 拆出的每个子任务（≤ 2 小时 / 任务）完成后。
> 目标：**每个任务跑完做 5 问小结，发现的通用知识沉淀到 knowledge-path.md**。

## 什么时候做

| 时机 | 是否必做 |
|---|---|
| 子任务 GREEN + REFACTOR + commit 后 | **必做** |
| 子任务中断 / 失败 | **必做**（失败复盘更重要）|
| brainstorming / spec / plan 阶段调整 | **必做**（写入对应的 spec / plan 文档的"复盘"小节）|
| 阶段 8 整体完成 | 选做（合并到阶段 8 复盘即可）|

## 5 问复盘模板

每个子任务完成后，写一段 ~ 200 字的复盘，5 个固定问题：

```markdown
## 复盘 {YYYY-MM-DD} 任务 {T-XX} {任务名}

> 提交：{commit hash}
> 用时：实际 {Xh} / 计划 {Yh}

1. **我做了什么？**
   {1-2 句话描述实际产出和计划是否一致}

2. **我跑偏了哪里？**
   {列出与原 plan 不一致的地方；没有写"无"}

3. **我发现什么通用知识？**
   {新学到的"下次还能用"的东西。可能是：字段用法、API 行为、库的特性、合规要求等}
   - 如果有，追加到 `knowledge-path.md` 引用的知识库
   - 如果无，写"无"

4. **我哪里慢了 / 卡了？**
   {用了比预期更多时间的部分；下次如何避免}

5. **下次会改什么？**
   {1-2 条对 plan / spec / process 的改进建议}
```

---

## 复盘的去向

| 复盘内容 | 去向 |
|---|---|
| 任务级复盘（5 问）| 写到子任务 commit message 或 PR 描述里 |
| 通用知识（5 问的 Q3）| 沉淀到 `knowledge-path.md` 指向的知识库（按主题归档）|
| 流程改进（5 问的 Q5）| 累计到阶段 8 完成后，写一个"阶段 8 整体复盘" |
| 跨阶段经验 | 写回 `references/` 里的方法论文档（贡献给 skill 本身）|

---

## 沉淀通用知识的具体做法

`Q3 我发现什么通用知识` 是最关键的一问。**没有沉淀 = 没复盘**。

### 什么算"通用知识"

满足以下任一条件：

| 条件 | 例子 |
|---|---|
| 同类项目都会遇到 | "WMOS 业务表都用 WM_VERSION_ID 做乐观锁" |
| 不在文档里 | "Oracle 的 NVL 在 PG 写 COALESCE；本项目统一用 NVL" |
| 容易踩坑 | "Spring Boot 的 @Transactional 在同类调用时不生效" |
| 跨表 / 跨模块共用 | "ASN 状态 30 → 40 必须配 PIX_TRAN_TYPE = RCV" |
| 跟合规相关 | "GSP 00201 要求所有批次可追溯 → 所有收货必须写 BATCH_NBR" |

### 怎么沉淀

1. **定位知识库**：从 `knowledge-path.md` 找到对应主题
2. **写新条目**：遵循知识库的现有格式（每条带日期 / 来源 / 验证状态）
3. **标记来源**：在条目末尾加 "来源：{项目名} 复盘 {YYYY-MM-DD}"
4. **跑 `field-alignment-check.py` 确认没破坏现有知识库一致性**

### 反例（不沉淀的）

- "今天加班了" — 个人感受，无普适性
- "代码改了 5 个文件" — 这是 commit 的事，不是知识
- "客户很难搞" — 不构成可复用的判断依据

## 阶段 8 整体复盘

阶段 8 全部子任务完成后，写一份**阶段级复盘**：

```markdown
# 阶段 8 整体复盘：{项目名}

> 完成时间：{日期}
> 总用时：实际 {X 天} / 计划 {Y 天}
> 子任务数：{N}（成功 {A} / 失败 / 重做 {B}）

## 整体评估

- 计划符合度：{百分比}（实际任务数 / 计划任务数）
- 用时偏差：{百分比}
- 回测通过率：{百分比}

## 流程改进（汇总子任务 Q5）

1. {...}
2. {...}

## 沉淀到知识库（汇总子任务 Q3）

1. {知识名}（已沉淀到 {路径}）
2. {...}

## 下次项目的改动建议

- 流程层：{...}
- 模板层：{...}
- 工具层：{...}
```

> **目的**：把单项目经验转化为 skill 本身的进化（通过 PR 贡献到 `references/` 或 `templates/`）。

---

## 复盘的反模式

| 反模式 | 怎么改 |
|---|---|
| 写"无 / 一切顺利" | 强制要求至少 1 条 Q3 或 Q5 |
| 写流水账（"我先做了 A，再做了 B"）| Q1 限定 1-2 句话；超过的部分归 commit message |
| 复盘但不沉淀 | 写完复盘 → 当场沉淀到知识库，**不要等** |
| 只写技术问题 | 流程 / 沟通 / 工具问题更要写（技术问题 commit 里能看到）|
| 复盘堆在脑子里 | 文字化。文字化才能 grep，才能跨项目用 |
