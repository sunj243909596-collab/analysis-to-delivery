# 设计回测（Design Backtest）

> 适用于阶段 8 完成 → 阶段 9（QA 审计）开始之间。
> 目标：**拿历史数据反向跑一遍新设计**，找出没考虑到的边界。

## 什么时候做

| 触发场景 | 必做 / 选做 | 位置 |
|---|---|---|
| 数据模型有新增 / 扩展表 | 必做 | 阶段 8.2 数据模型设计后 |
| 业务规则新增 / 修改 | 必做 | 阶段 8.1 FSD 后 |
| 状态机新增 / 修改 | 必做 | 阶段 8.1 FSD 后 |
| 字段对齐有问题 | 必做 | `field-alignment-check.py` 报错时 |
| 合规相关变更（GSP / HIPAA / SOX）| 必做 | 阶段 3 合规评审后 |
| 纯前端 UI 调整 | 选做 | - |
| 配置 / 文案修改 | 选做 | - |

## 回测 4 大类

按重要性倒序：

### 1. 数据模型回测（最关键）

**目的**：验证新表 / 新字段 / 新索引在历史数据上能正确工作。

**步骤**：
1. 从 `knowledge-path.md` 找到生产数据源（测试库 / 影子库）
2. 抽取最近 6-12 个月的真实数据样本（脱敏）
3. 在样本上跑：
   - 新增表的 DDL — 是否能成功创建
   - 索引 — 是否有合适的命中
   - 查询 — 用新查询拿真实数据看是否返回正确
4. 写回测报告（见下文模板）

**工具**：
- Oracle 库用 `oracle-wms19tst` skill
- PG 库用对应 `*-wms19tst` 或 `pg_*` skill
- 无测试库时跳过但**必须记录原因**

### 2. 业务规则回测

**目的**：验证新业务规则不会破坏已有业务流。

**步骤**：
1. 列出新规则的 3-5 个关键场景（来自 BRD / FSD）
2. 对每个场景，找历史数据中的对应样本（10-50 笔）
3. 手工按新规则跑一遍，看结果是否与预期一致
4. 特别关注**异常路径**（拒收 / 取消 / 锁定 / 解锁）

**输出**：
```markdown
| 场景 | 样本量 | 命中 | 异常 | 备注 |
|---|---|---|---|---|
| 收货正常流 | 30 笔 | 30 | 0 | |
| 收货拒收 | 5 笔 | 4 | 1 | 样本 WMS-2026-001 状态机未触发（待排查）|
```

### 3. 状态机回测

**目的**：验证新增 / 修改的状态不会产生非法跃迁。

**步骤**：
1. 画出状态机 ASCII 图（参考 `flow-chart-ascii.md`）
2. 列出所有合法跃迁
3. 抽取 100 笔历史单据，**回放**状态变化历史
4. 比对：实际跃迁 ⊆ 合法跃迁？

**注意**：
- 状态机回测 ≠ 业务规则回测：前者看"状态能不能这么变"，后者看"业务上该不该这么变"
- 状态机错通常是代码 bug；业务规则错通常是设计 bug
- 两者必须分开做

### 4. 字段对齐回测

**目的**：确保文档里写的字段名 / 类型 / 长度与生产库一致。

**步骤**：
1. 跑 `python3 scripts/field-alignment-check.py` 自动检查
2. 对 P1 / P0 项，人工到 `oracle-wms19tst` 库确认
3. 自创字段（文档有、库无）— 必须确认是有意新增
4. 库有、文档无（废弃字段）— 确认是否真废弃

## 跳过回测的合法理由

记录在 `06-开发设计说明书.md` 的"九、设计回测"小节：

| 理由 | 怎么做 |
|---|---|
| 无测试库 | 写明"无测试库，跳过数据回测" |
| 纯前端调整 | 直接跳过全部 |
| 紧急 hotfix | 事后补；hotfix PR 描述里写"回测后补" |
| 项目全新无历史数据 | 跑 mock 数据（参考 `07-测试用例设计.md` 里的 mock 数据）|

## 与 superpowers 的对应

| superpowers 概念 | 本 skill 的落地 |
|---|---|
| `verification-before-completion` 铁律 | 阶段 8 收尾前必须跑回测，否则不进阶段 9 |

| `using-superpowers` 提示 | 任何回测失败 → 回到 brainstorming / spec 重新审视 |
