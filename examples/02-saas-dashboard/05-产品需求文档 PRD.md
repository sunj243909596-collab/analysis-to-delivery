# 产品需求文档（PRD）：SaaS 后台 — 客户订单管理

> 状态：✅ 已确认

## 一、产品概述

### 1.1 背景
B2B 中小企业当前用 Excel + 微信群管理订单,效率低(30 分钟/单)、错误率高(5%)、无审计、缺统计。详见 `01-业务需求文档 BRD.md` §一。

### 1.2 目标
- 订单全流程在线化(下单 → 支付 → 发货 → 完成)
- 处理效率提升 50%(30 分钟 → 15 分钟)
- 错误率降至 0.5% 以下
- 多租户数据隔离 100%
- 数据看板 T+1 准确率 100%
- GDPR/PIPL 轻度合规

### 1.3 范围
**包含**：客户/商品/订单基础管理、订单全流程、支付集成(支付宝/微信/Stripe)、物流跟踪(顺丰)、数据看板、多租户隔离。
**不包含**：财务记账(用现有 ERP)、仓储管理(用现有 WMS)、移动端 APP。

## 二、用户故事

| 角色 | 场景 | 需求 | 优先级 |
|---|---|---|---|
| 客户(下单人) | 注册后下单 | 注册/登录 → 浏览商品 → 加购物车 → 提交订单 → 支付 → 查物流 → 确认收货/申请退款 | P0 |
| 销售(跟单员) | 审核订单 | 收到待审核 → 审核订单/确认收款 → 安排发货 → 处理退款申请 | P0 |
| 仓管 | 拣货发货 | 收到发货任务 → 拣货 → 出库 → 更新物流 → 标记已发货 | P0 |
| 管理员 | 客户管理/权限 | 创建客户账号 → 分配角色 → 维护商品/价格策略 | P1 |
| 数据看板 | 看销售统计 | 看日/周/月销售 → 按客户/商品分组 → 导出 | P2 |

## 三、功能需求

### 3.1 客户下单

#### 3.1.1 功能说明
客户在 Web 端选商品加购物车 → 提交订单 → 系统校验库存 → 锁定库存 → 生成订单 → 推送支付链接。

#### 3.1.2 业务流程
详见 BRD §三.3.1 泳道图 + `业务流程图-订单创建.txt`。

#### 3.1.3 界面设计
（响应式 Web,React 19 + TS;移动端响应式;6 步骤结账向导：购物车 → 地址 → 支付方式 → 确认 → 支付 → 完成）

#### 3.1.4 字段定义

| 字段名 | 类型 | 必填 | 校验 | 数据来源 | 备注 |
|---|---|---|---|---|---|
| order_no | VARCHAR(20) | 是 | 唯一(UUID 简化) | 系统生成 | |
| tenant_id | UUID | 是 | 非空 + 用户所属租户 | JWT | 多租户隔离 |
| customer_id | UUID | 是 | 所属租户 | customers | |
| status | VARCHAR(20) | 是 | 枚举(draft/submitted/paid/shipped/completed/refunded/cancelled) | 状态机 | 字符串枚举 |
| subtotal | NUMERIC(12,2) | 是 | > 0,保留 2 位 | 计算 | |
| version | INTEGER | 是 | ≥ 0(乐观锁) | 自增 | |
| created_at | TIMESTAMPTZ | 是 | 默认 NOW() | DB | |
| is_deleted | BOOLEAN | 是 | 默认 false | DB | 软删除 |

#### 3.1.5 业务规则
- 库存不足 → 阻止下单,提示"库存仅剩 X 件"
- 并发超卖 → DB 悲观锁(`SELECT ... FOR UPDATE`)+ 乐观锁(`version`)组合
- 订单状态机:draft → submitted → paid → shipped → completed;允许 cancelled(草稿/已提交)、refunded(已支付/已发货)
- 支付回调幂等性:同一 `payment_id` 多次回调,只生效一次(用 Redis SETNX)
- 多租户隔离:所有查询强制带 `tenant_id`(中间件 + Prisma 全局 scope)

#### 3.1.6 异常处理

| 异常 | 提示 | 处理 |
|---|---|---|
| 库存不足 | "库存仅剩 X 件,请调整数量" | 阻止下单 |
| 支付失败 | "支付失败,请重试或更换支付方式" | 订单保持 submitted,允许重新支付 |
| 支付超时 | "支付超时(15 分钟),订单已取消" | 释放库存,订单 → cancelled |
| 网络中断 | "网络异常,请重试" | 客户端重试 + 服务端幂等 |
| 多租户越权 | "无权访问该订单" | 403 拒绝 + 记审计日志 |

#### 3.1.7 权限控制
- 客户：可创建/查看/取消自己的订单,可支付/申请退款
- 销售：可审核/确认收款/安排发货,可看所辖客户订单
- 仓管：可改订单状态(拣货 → 发货)、可更新物流
- 管理员：可分配权限、不可改订单金额
- 数据看板：只读统计

### 3.2 支付集成

#### 3.2.1 功能说明
订单提交后 → 系统生成支付链接 → 客户扫码/跳转支付 → 支付回调 → 订单状态 paid。

#### 3.2.2 业务规则
- 支付链接有效期 15 分钟
- 支付回调必须验证签名(支付宝/微信/Stripe 各自签名机制)
- 幂等性:同 `payment_id` 多次回调只生效一次
- 降级:支付通道故障时,允许销售手工"确认收款"(限管理员/销售主管)

#### 3.2.3 字段定义

| 字段名 | 类型 | 必填 | 校验 | 数据来源 |
|---|---|---|---|---|
| payment_id | VARCHAR(40) | 是 | 唯一 | 支付通道 |
| order_no | VARCHAR(20) | 是 | orders 存在 | orders |
| channel | VARCHAR(20) | 是 | 枚举(alipay/wechat/stripe/manual) | 输入 |
| amount | NUMERIC(12,2) | 是 | > 0,= order.subtotal | 计算 |
| status | VARCHAR(20) | 是 | 枚举(pending/success/failed) | 支付回调 |
| paid_at | TIMESTAMPTZ | 条件 | status=success 时必填 | 支付回调 |

#### 3.2.4 权限控制
- 支付通道 webhook:系统角色,无人手工调用
- 手工确认收款:仅管理员/销售主管

## 四、非功能需求

### 4.1 性能
- 单订单创建响应 < 500ms(P95)
- 并发下单:100 单/秒(峰值)
- 数据看板 T+1 加载 < 3s
- 支付回调处理 < 200ms(P95)

### 4.2 安全
- HTTPS 强制(TLS 1.3)
- 密码 bcrypt(cost=12)
- 多租户行级隔离(应用层中间件 + Prisma 全局 scope)
- SQL 注入防护(Prisma ORM 参数化)
- XSS 防护(React 默认转义 + CSP)
- CSRF Token(关键操作)
- 审计日志完整(所有改/查操作记录到 `data_access_log`)

### 4.3 兼容性
- Web:Chrome 100+ / Edge 100+ / Safari 16+
- 移动端:响应式,iOS Safari 16+ / Android Chrome 100+
- 后端:Node 22 / Express 5 / PostgreSQL 16 / Prisma 5
- 区域:EU 客户数据存 EU 区域(GDPR 要求)
